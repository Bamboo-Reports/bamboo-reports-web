# **5\. Naming and URL structure: the /gcc/ silo**

* Put every GCC page under one /gcc/ silo, with /gcc/ itself as the GCC Tracker directory. This is the chosen structure after evaluating the options (flat root slugs, a subdomain, and a product-path scheme).

## **The scheme**

| Page | URL | Notes |
| :---- | :---- | :---- |
| Directory / pillar | /gcc/ | The GCC Tracker. Targets 'list of GCCs in India', 'GCC tracker', 'GCC database India'. Branded 'GCC Tracker by Bamboo Reports'. |
| Company | /gcc/companies/{company} | e.g. /gcc/companies/3m-co |
| City | /gcc/cities/{city} | e.g. /gcc/cities/bengaluru |
| Sector | /gcc/industries/{sector} | e.g. /gcc/industries/bfsi |
| GCC type | /gcc/types/{type} | e.g. /gcc/types/rnd |
| Service | /gcc/services/{service} | e.g. /gcc/services/finance-accounting |
| Insights / listicles | /gcc/insights/{slug} | News, interviews, guides. Inside the silo on purpose. |
| Vanity | /gcc-tracker | 301-redirect to /gcc/. |

## **Why this is best for the goal**

* Topical authority: one tight, self-contained GCC section is the strongest signal that Bamboo Reports is the GCC authority, which is the goal.

* Keyword in every path without stuffing, so we get the relevance and click-through benefit. The exact query phrase lives in the title tag and H1; the URL just needs to be clean and contain the keyword.

* All authority accrues to the bambooreports.com root, so the brand surfaces for everything (a subdomain would split that authority, which is why we did not use one).

* Spinoff stays a one-step move: because it is a single subtree, later we can 301 /gcc/\* to gcctracker.com/\* and the directory becomes gcctracker.com's home. Keep the option, do it only once gcctracker can stand alone.

## **Rules**

* Slugs are readable keyword slugs (3m-co, bengaluru, bfsi), lowercase and hyphenated. Namespaced by type (companies, cities, industries, types, services) so slugs never collide.

* Set slugs once and never change them; changing URLs later forfeits rankings. Build /gcc/ self-contained (own sitemap, internal-link cluster, templates) so a future move is clean.

# **8\. The page template and the 10% data rule**

* The 3M page (companion file) is the template. Each page shows about 10% of the record: identity, verified counts and ranges, one Bamboo insight, and a last-updated line. Everything per-center, per-service and per-contact is a locked count that drives registration. Full field split is in the gating matrix (companion file).

* Never on a free page: exact addresses, per-center headcount, contact emails, named software, the filtered-list export, or the \~150 unique companies.

## **Two fixes on the current template before scaling**

* Contact names must not sit in the page source. CSS blur is cosmetic; the name is still in the HTML for anyone who views source and for crawlers. Show titles only, with no surname in the markup.

* Remove the preview password gate for production. Public pages must be fully static and crawlable, with real internal routes. Keep what the template gets right: brand blue and logo, human copy with no em dashes, the four schema blocks, the four CTAs, the labelled sponsor slot.

# **10\. How to build a page (the process)**

1. One template per page type, parameterised. Generate pages from the warehouse as static or incrementally-static HTML (SSG/ISR) so they are fully crawlable. The gated and live parts (counts, contact reveals) come from the per-view server-side endpoints.

2. For each entity: pull the record, compute the verified counts and the \~10% visible fields per the gating matrix, and fill the template.

3. Auto-generate the title, meta, H1, canonical and schema from the data.

4. Write the unique insight line and the short editorial. This is the anti-thin-content step; a page that is only a stat block will not rank and risks the scaled-content filter.

5. Apply the gating: contact titles only (no names in source), locked teasers for gated fields, exclude the \~150 unique companies.

6. Add internal links to the parent hubs and siblings, and the breadcrumb.

7. Run the per-page QA checklist (section 18\) and validate the schema in Google's Rich Results Test.

8. Add the URL to the /gcc sitemap and publish it in the wave batch, not all at once.

# **12\. The freemium subscription (the core revenue)**

| Tier | Price | What they get |
| :---- | :---- | :---- |
| Free | $0 | Browse and filter all \~2,668 to counts; a static sample of \~20 companies at fuller depth; teaser counts everywhere; a few credits to reveal a handful of contacts. No export. |
| Individual | \~$2-3k/yr, 1 seat | Full company and center detail; contact reveals metered by credits; no bulk export. |

* **The fence:** filter to a full list plus export is Enterprise only. Free sees counts, Individual sees capped records, Enterprise gets the set. The credits and caps are the conversion mechanism, tuned below the smallest Enterprise deal so they cannot rebuild a list.

# **15\. SEO execution**

* Static or server-rendered HTML; unique title, meta and H1; real canonical; robots index and follow; every URL in the /gcc sitemap submitted to Search Console and Bing; fast and mobile-first; genuine unique content; internal links to hubs. Match the depth of the pages that rank, then beat them on accuracy.

* **Backlinks and digital PR are the biggest lever and the one we lack most.** Turn verified stats and interviews into press and earn links to bambooreports.com. E-E-A-T: named analysts, a stated methodology, a 'verified' label.

# **16\. AEO execution (AI answers)**

* We are invisible in AI answers today, so this is deliberate. On page: schema with dateModified, sameAs and the verified counts; facts as plain sentences AI can lift; a visible last-updated and sources line. Off page: get Bamboo into the third-party 'GCC providers' and 'Top 10' articles that already rank and get cited; run data-led PR; create one Wikidata entity for Bamboo Reports with GCC Tracker as its product; publish an llms.txt at the root. The /gcc/insights engine is the strongest AEO asset.

# **17\. The technical prerequisite**

* Nothing here is safe until the data is served correctly. The migration to per-view server-side endpoints must enforce entitlement per endpoint (plan checked server-side; only allowed fields, rows and limits returned), put hard caps on the filter and list endpoints (no 'return all matching' below Enterprise), and fix the separate profiles-table leak with its own row-level security rule (auth.uid() \= user\_id). Recover speed with caching: the free and public data is identical for everyone, so cache counts and aggregates hard and at the CDN, and compute only the per-user parts live. These endpoints also feed the public pages.

## **Per-page QA checklist (before publish)**

* Unique title, meta, H1 from real data; real canonical; robots index,follow; in the /gcc sitemap; real verified figures; no placeholder text or em dashes; contact titles only (no names in source); gated data as locked counts; \~150 unique companies excluded; all four schema blocks passing Rich Results Test with dateModified equal to the visible date; internal links to hubs and a breadcrumb; fully static and crawlable; four CTAs present and ?src= tagged; sponsor slot, if used, labelled and rel=sponsored.

# **19\. Measurement**

* Search: rankings on target company, city, sector and head queries; indexed pages; AI citations and AI-referral traffic; referring domains.

* Freemium funnel: organic sessions, then free registrations, then activated free users (used credits or filters), then Individual conversions, then Custom leads (consult bookings), plus Q1 report downloads and newsletter subscribers. Track registration and conversion rate by page type.

* Monetization: sponsor inventory sold and advertorial revenue, separate from subscription revenue.

# **20\. Do-not-do**

* Do not dump thin pages at scale; run gcctracker as a separate site now; use a subdomain (it splits authority); put ads on the product; publish the \~150 unique companies, exact addresses or contact emails; leave contact names in the page source; render content only in JavaScript; change slugs once set; or ship placeholder text or em dashes.
